补充说明：当前实际开发环境为 Windows 下的 PowerShell（pwsh）终端，因此本文中涉及命令行、脚本入口和环境兼容性的建议，均默认以 PowerShell 作为执行环境。

## 2. 需要优先修改的问题

### P0. 任务口径有偏差：把“陈述真伪分类”和“开放式生成幻觉检测”混写了

问题：文档开头写的是“检测模型生成内容中的幻觉”，但课程文档与当前基础实验的主线其实是基于 True-False Dataset 对给定陈述做真假判别，而不是先生成答案再检测幻觉。这个表述会导致后续方法设计和实验对象发生漂移。

影响：

- 容易把基础任务误做成生成式 QA 幻觉检测。
- 会让 Phase 4 的注意力设计直接套用 query-answer 结构，而忽略当前数据是陈述句。
- 报告中“任务定义”与“实验实现”可能不一致。

建议：把项目目标统一改成“对给定陈述的真伪进行判别，并分析模型内部是否编码真实性信号”。如果后续要做开放式生成幻觉检测，应明确标成 3.3 的扩展方向，而不是基础任务默认设定。

建议替换文案：

```markdown
本项目聚焦于利用大语言模型的内部状态判断给定陈述的真伪，并分析模型内部是否编码了与真实性相关的表征信号。实验数据以 True-False Dataset 中的陈述句为主，基础任务不以开放式问答生成作为主要评价对象。核心任务包括：

1. 实现并对比基于序列概率/困惑度的方法与基于隐藏状态分类的 SAPLMA 方法；
2. 分析不同层深度、不同 token 表示方式对检测性能的影响；
3. 在上述基线之上探索注意力或 FFN 激活等内部特征，提出并评估改进方法。
```

### P0. 主模型可行性写得过于绝对，和后文风险分析不一致

问题：计划把 Llama-2-7B 4-bit 写成“8GB 显存下可运行”的主方案，但后文又明确写了 Windows 下 bitsandbytes 兼容性风险高、4-bit 也可能 OOM。对于需要输出 hidden states、attentions、FFN activations 的实验，实际显存压力会明显高于单纯文本生成。

影响：

- 容易让计划显得“理论可行、实际难落地”。
- 若后续实际改用 Qwen2-1.5B，计划与实现会出现偏差。
- 进阶实验尤其容易因为显存或兼容性问题卡住。

建议：不要把 Llama-2-7B 4-bit 写成默认必成方案。更稳妥的写法是：优先尝试 Llama-2-7B 4-bit；若 Windows + 8GB 环境下运行不稳定，则以 Qwen2-1.5B 作为主实验模型完成全流程，并在条件允许时用 Llama 做补充验证。

建议替换文案：

```markdown
| 项             | 选择                                                                  | 说明                                                  |
| -------------- | --------------------------------------------------------------------- | ----------------------------------------------------- |
| 优先尝试模型   | Llama-2-7B（4-bit 量化）                                              | 课程推荐模型；若环境稳定，优先用于主实验              |
| 保底主实验模型 | Qwen2-1.5B（FP16）                                                    | 在 Windows + 8GB 显存环境下更稳妥，可完整完成课程要求 |
| 使用策略       | 先完成 Qwen2-1.5B 端到端跑通，再尝试将核心实验迁移到 Llama-2-7B 4-bit | 降低环境风险                                          |
```

### P0. Phase 4 方向 A 与当前数据集不完全匹配

问题：方向 A 中使用了“answer tokens 对 query tokens 的注意力占比”这一设计，但当前必做数据集是 True-False Dataset，基础输入是陈述句，并不天然具有 query-answer 结构。

影响：

- 如果不引入额外 QA 数据集，这个特征定义无法直接落地。
- 会造成 3.3 的创新方向看起来合理，但实现时缺少明确输入格式。

建议：二选一。

1. 如果坚持使用 True-False Dataset，就把注意力特征改写成陈述句内部的实体-关系注意力特征，例如主语 token、关系词 token、句尾 token 之间的注意力统计。
2. 如果坚持使用 query-answer attention，就在计划中明确说明 3.3 会额外引入 QA 数据集，并把数据准备、提示模板和生成流程写进去。

建议替换文案：

```markdown
| P4.A2 | 计算主语实体 token、关系词 token 与句尾 token 之间的注意力统计特征 | attention features |
```

如果要保留 query-answer 设计，建议额外补一句：

```markdown
该方向将额外引入问答型数据集，并通过统一提示模板生成 answer，再计算 answer token 对 query token 的注意力特征。
```

### P0. 实验协议不够明确，后续很难保证结果可比与可复现

问题：文档写了“划分 train/val/test 并预处理”，但没有说明按什么策略切分、是否分层抽样、真假样本是否成对保留、是否固定随机种子、是否做多次重复实验。

影响：

- 结果复现困难。
- 不同方法之间可能出现数据泄漏或分布不一致。
- 报告中的图表和结论可信度会被削弱。

建议：在计划里新增一个“实验协议”小节，至少明确以下内容：

- 划分策略：按领域分层随机划分，建议 8:1:1。
- 成对约束：真/假对应陈述若存在配对关系，必须整体划入同一集合，避免泄漏。
- 随机种子：固定 seed，例如 42。
- 指标：Accuracy、Macro-F1、AUROC。
- 重复次数：建议 3 个随机种子，报告均值和标准差。

建议新增文案：

```markdown
### 实验协议

- 数据划分采用分层随机划分，训练集/验证集/测试集比例为 8:1:1；
- 若数据中真/假陈述存在配对关系，则同一对样本必须划入同一集合，避免信息泄漏；
- 所有分类实验固定随机种子，并建议在 3 个随机种子下重复实验，报告均值与标准差；
- 统一汇报 Accuracy、Macro-F1 与 AUROC，确保不同方法之间可比。
```

## 3. 建议同步修订的问题

### P1. Windows PowerShell 环境与 .sh 脚本命名冲突

问题：文档前面明确写的是 Windows 原生环境；结合当前实际使用的是 PowerShell（pwsh），项目结构中的运行脚本若继续使用 .sh，会与默认执行环境不一致。除非你们明确依赖 Git Bash 或 WSL，否则这一点应在计划中修正。

建议：

- 既然当前环境是 PowerShell，优先将运行脚本改为 .ps1；或
- 用 Python 入口脚本统一执行，例如 scripts/run_baseline.py。

建议替换文案：

```markdown
├── scripts/
│ ├── run_baseline.ps1 # Windows 下运行基础方法
│ ├── run_analysis.ps1 # Windows 下运行分析实验
│ └── run_advanced.ps1 # Windows 下运行进阶实验
```

如果你们确实打算用 Git Bash/WSL，也应在环境部分明确写出依赖条件；否则 review 建议默认以 PowerShell 脚本作为推荐形式。

### P1. “0-31 层”写死了模型结构，不利于 Llama/Qwen 双模型兼容

问题：Phase 3 直接写“0-31 层”，默认绑定了 Llama-2-7B 的层数。如果后续因为环境问题切换到 Qwen2-1.5B，这个表述就会失效。

建议：统一改成“遍历模型全部 Transformer 层”，并在实现层面使用 model.config.num_hidden_layers；同时说明是否将 embedding output 记作第 0 层，避免索引歧义。

建议替换文案：

```markdown
| P3.1 | 提取模型全部 Transformer 层的隐藏状态 | 分层特征矩阵 | B |
```

并补充一句：

```markdown
层分析统一以 Transformer block 输出为统计对象，不将 embedding output 计入层号。
```

### P1. PPL 方法的命名和评分口径不够清楚

问题：伪代码函数名叫 compute_statement_probability，但返回值其实是 perplexity。概率、负对数似然和困惑度不是同一个量，如果文档中混用这些词，后续报告会不够严谨。

建议：统一把 Phase 2 的基线命名为“基于序列概率/困惑度的方法”，并在实现细节里明确采用哪一个实际评分。更稳妥的写法是使用平均负对数似然或 PPL，并强调做长度归一化。

建议替换文案：

```markdown
def compute_statement_ppl(model, tokenizer, statement):
"""计算陈述句的长度归一化困惑度，用作真伪判别分数。"""
```

### P1. Token 位置分析里的 Subject token 定义不清楚

问题：文档提出比较 Subject token(s)，但没有说明如何从 True-False Dataset 的陈述句中定位主语实体范围。

建议：在计划中明确 Subject token 的提取方式。可选方案有两种：

1. 手工规则：将句首第一个命名实体或名词短语作为主语。
2. 自动解析：使用 NER 或依存句法抽取主语实体。

如果时间有限，建议把 Subject token 分析设为可选项，最低保底先比较 First token、Last token 和 Mean pooling。

## 4. 建议弱化或改写的目标表述

### M2 和 M4 不建议写成硬性准确率承诺

问题：

- M2 写成“>60% 准确率”；
- M4 写成“至少提升 3-5% 准确率”。

这两个目标都不是课程文档的硬要求，而且容易受到模型规模、切分策略、量化方式和随机种子的影响。计划里写成硬门槛，后续如果未达到，反而显得计划“失败”，即使实验本身和分析都是合格的。

建议：改成“完成可复现实验并给出充分分析”，把性能提升写成期望而不是验收门槛。

建议替换文案：

```markdown
**里程碑 M2**：完成 PPL 与 SAPLMA 两类基线方法的实现、验证和对比，获得可复现的验证集结果，并输出 Accuracy、Macro-F1 与 AUROC。

**里程碑 M4**：至少完成一种进阶特征的实现与对比实验；若性能未超过基线，也需分析失败原因、适用条件和观察到的内部模式差异。
```

### 时间安排建议改成相对周次或更新为当前版本日期

问题：当前计划使用的是固定日历日期。如果这份文档是后续继续使用的执行计划，那么日期很快会过时。

建议：

- 如果这是课程立项时的存档版本，可以保留；
- 如果这是实际执行版本，建议改成“第 1 周到第 5 周”或直接更新到当前时间表。

## 5. 建议补充的内容

### 建议补充 1. 最小可交付版本定义

为了确保课程要求一定能完成，建议在计划中明确“最低保底版本”：

- 使用 Qwen2-1.5B 完成 PPL、SAPLMA、层分析、token 位置分析；
- 至少完成一个进阶方向的小规模实验；
- 输出代码、图表、实验记录和报告初稿。

这样即使 Llama-2-7B 或注意力/FFN 方向受环境限制，也不会影响课程最低要求的完成。

### 建议补充 2. 报告输出物和图表清单

课程文档强调 3.1 和 3.2 的详尽图表与分析。建议在计划中预先定义图表清单，例如：

- PPL vs SAPLMA 对比表；
- 层深度 vs Accuracy 曲线；
- 不同 token 表示方式的柱状图；
- 进阶方法与基线的对比表；
- 失败案例或错误类型分析表。

这样能让 Phase 5 更有落点，而不是只写“整理图表”。

## 6. 推荐采纳顺序

建议按以下顺序修改原计划文档：

1. 先统一任务定义，把“真假陈述分类”作为主线写清楚。
2. 再调整模型策略，把 Qwen2-1.5B 明确成保底主实验模型。
3. 补上实验协议与复现规范。
4. 修正 Phase 4 中与数据集不匹配的 attention 特征定义。
5. 最后统一脚本形式、里程碑口径和时间表表述。

这份计划的框架是对的，课程要求也基本覆盖到了；真正需要修的是任务定义、环境可行性和实验协议这三处。只要把这三处改稳，整份计划就会从“能看”变成“能执行”。
