# Project_Plan 审查与修改建议（v4）

## 1. 审查结论

经过前几轮修订，当前的 [docs/Project_Plan.md](docs/Project_Plan.md) 已经没有明显的结构性偏差，课程要求覆盖也基本完整。和 v1、v2、v3 相比，这一版已经可以视作“接近执行版”的计划文档。

本轮 v4 不再处理大框架问题，而是继续检查剩余的实验闭环、PowerShell 执行一致性和技术表述准确性。结论是：文档已经足以指导实现，但仍有 4 处值得再收紧的细节。如果这些点也补齐，后续计划文档层面的审查基本就可以结束，重心可以转向代码与实验实现。

## 2. 本轮重点

本轮审查主要围绕以下三个问题展开：

1. 实验协议是否已经足够明确到可以避免后续数据泄漏或结果口径混乱；
2. PowerShell 环境下的操作说明是否真的可以直接执行；
3. 显存预算和特征抽取规则是否存在容易误导实现者的表述。

## 3. 剩余问题与修改建议

### P1. 随机种子与数据划分的边界仍不够明确

当前计划已经补充了 train / val / test 的职责划分，这是正确的；但随机种子这一条仍然写成：

```markdown
所有分类实验固定随机种子（默认 42），建议在 3 个随机种子（42, 123, 2024）下重复实验，报告均值与标准差
```

问题在于，这句没有明确说明：

- 这 3 个种子是否会改变数据划分；
- 还是只改变分类器初始化、训练顺序和抽样顺序；
- 当 [docs/Project_Plan.md](docs/Project_Plan.md) 中的 M3 明确说“层深度和 token 位置分析将为进阶实验提供方向”时，Phase 3 若参与后续特征选择，应基于验证集还是测试集来做决策。

如果这一点不补清楚，后续可能出现两种常见问题：

- 不同随机种子对应不同数据切分，导致方法间可比性下降；
- 用测试集上的层分析 / token 分析结果来选后续实验策略，形成隐性测试集泄漏。

建议：把“数据划分种子”和“模型/分类器训练种子”分开写；同时明确，若 Phase 3 的分析结论会用于指导 Phase 4 的特征选择，则该决策应只基于训练集+验证集完成，测试集只保留给最终固定方案。

建议新增文案：

```markdown
- **种子约定**：数据划分使用固定 split seed（建议 42）一次生成并全程复用；42、123、2024 这 3 个随机种子仅用于分类器训练、阈值调优过程中的随机性控制，不重新划分数据集。
- **分析使用边界**：若层深度分析或 token 位置分析的结论将用于后续方法选择，则该选择过程仅基于训练集与验证集完成；测试集只用于固定方案后的最终结果汇报。
```

### P1. Subject token 的默认抽取策略仍然偏向命名实体，不适合全部数据域

当前文档把 Subject token 的默认方案写成“优先使用 NER 工具（如 spaCy）自动抽取句首命名实体”。这个策略对城市、公司等领域可能有效，但对 animals、scientific facts、chemical elements 这类包含大量普通名词短语的陈述并不稳妥。

例如类似“Blue whales are mammals”或“Helium is a noble gas”这类句子里，主语并不一定会被稳定识别为命名实体。于是会出现：

- Subject token 分析对不同领域的成功率不一致；
- 后续注意力特征中“主语实体”定义不稳定；
- 计划里写的是统一流程，实际执行时却会退化成大量人工处理。

建议：把 NER 从“优先默认策略”调整为“可选辅助策略”，把更通用的名词短语 / 依存主语抽取作为第一选择。若你们仍想保留 spaCy，也建议明确它的主要作用应是句法分析或 noun chunk 提取，而不仅仅是 NER。

建议替换文案：

```markdown
- **Subject token 提取**：若实验中需要定位主语实体，优先使用依存句法或 noun chunk 规则抽取句首主语短语；对于能够稳定识别的命名实体样本，可辅以 NER 工具（如 spaCy）做对齐检查。若自动解析不可靠，则退化为手工规则，并仅在报告中标注结果。Subject token 分析作为可选项，保底至少比较 First token、Last token 与 Mean pooling。
```

### P2. PowerShell 环境下的环境变量示例仍然不是可直接执行的写法

文档已经明确当前环境是 Windows + PowerShell，但在模型下载部分的注意事项里，环境变量仍然用的是通用记法：

```text
HF_HOME=...
HF_ENDPOINT=...
```

这不算知识性错误，但放在 PowerShell 语境下不够直接。实际执行时，更合适的写法应该是 PowerShell 语法，例如：

```powershell
$env:HF_HOME = "F:\PythonCodes\LLMHallucinationProbing\models_cache"
$env:HF_ENDPOINT = "https://hf-mirror.com"
```

如果想写成持久化环境变量，也应单独说明 `setx` 或系统环境变量设置方式。

建议：在该小节中直接把 PowerShell 版示例写出来，避免文档一边强调 PowerShell，一边给出 shell-agnostic 记法。

建议替换文案：

```powershell
# PowerShell 临时设置（当前终端会话生效）
$env:HF_HOME = "F:\PythonCodes\LLMHallucinationProbing\models_cache"
$env:HF_ENDPOINT = "https://hf-mirror.com"
```

### P2. 显存预算表中仍有两处表述偏乐观或与主线任务无关

当前显存预算表和结论已经比早期版本稳很多，但还有两处措辞值得收紧：

1. [docs/Project_Plan.md](docs/Project_Plan.md) 中 Llama-2-7B 4-bit 一行仍写成“推荐，留有充足余量”。这个说法对普通生成推理还勉强成立，但对你们当前计划里的 hidden states、attention、FFN activation 提取来说并不稳妥。文档后面的风险表也已经承认 4-bit 下依然可能 OOM，因此“留有充足余量”过于乐观。
2. Qwen2-1.5B FP16（含梯度）一行写成“可进行微调实验”。这不仅超出了当前课程任务主线，而且在 8GB 环境下也容易被理解成“可以稳定进行完整微调”，这在多数情况下并不准确。

建议：

- 把 Llama-2-7B 4-bit 的描述改成“可尝试，但在提取 hidden states / attention 时余量有限”；
- 删除“可进行微调实验”这类与当前任务无关、且容易引发误解的表述，或至少改成“适合小 batch 的特征提取与参数高效实验”。

建议替换文案：

```markdown
| Llama-2-7B | **4-bit (NF4)** | **~4-5 GB** | 可尝试，但在提取 hidden states / attention 时余量有限 |
| Qwen2-1.5B | FP16 (含梯度) | ~6 GB | 适合小 batch 的特征提取或参数高效实验 |
```

## 4. 修改优先级建议

如果只再做一轮很小的修订，建议按以下顺序处理：

1. 先补清 split seed 与训练 seed 的区别，并明确 Phase 3 结论用于 Phase 4 时的验证集边界；
2. 再把 Subject token 的默认抽取策略从“NER 优先”改成“句法 / 名词短语优先”；
3. 接着把 PowerShell 环境变量示例改成可直接执行的语法；
4. 最后收紧显存预算表中的乐观措辞。

## 5. 一句话总结

当前计划文档已经足够接近最终执行版；v4 剩下的问题主要是“别让实现者被文字误导”。把随机种子边界、Subject token 策略、PowerShell 环境变量示例和显存表述再修一轮，这份计划文档基本就可以收口了。
