# Project_Plan 审查与修改建议（v3）

## 1. 审查结论

在吸收 v1 和 v2 的主要修改建议后，当前的 [docs/Project_Plan.md](docs/Project_Plan.md) 已经接近可执行版计划文档：

- 课程要求覆盖已经基本完整；
- 保底模型、PowerShell 环境和阶段目标的口径已经明显更稳定；
- Phase 4 的注意力方向也已经与 True-False Dataset 的陈述句格式做了初步对齐。

因此，本轮 v3 的重点不再是修正大方向，而是继续收紧实验闭环、环境执行一致性和复现边界。当前剩余问题数量已经不多，但如果要把这份文档作为真正的团队执行计划使用，仍建议补完以下 4 处细节。

## 2. 本轮重点

本轮审查主要关注以下三个方面：

1. 训练集、验证集、测试集在实验流程中的职责是否已经闭环；
2. Windows + PowerShell 环境下的可执行步骤是否前后一致；
3. 进阶方法与复现要求是否已经具备足够明确的落地说明。

## 3. 剩余问题与修改建议

### P1. 验证集与测试集的角色还需要进一步写清楚

当前文档已经在实验协议中定义了 train / val / test 三个集合，也在 Phase 1 中安排了数据划分；但在基础方法和里程碑表述中，仍主要强调“验证集评估”和“验证集结果”。这会留下一个执行层面的空白：最终图表和主结果到底应以验证集还是测试集为准。

这不是结构性错误，但如果不写清楚，后续非常容易出现以下问题：

- 在验证集上调完阈值后直接把验证集结果写进主结果表；
- 不同方法的模型选择与最终评估边界混在一起；
- 报告中测试集的角色被弱化，影响结果可信度。

建议：在实验协议中显式补一句“验证集只用于模型选择、阈值调优和早停，测试集只用于最终一次汇报”；并在 Phase 2、Phase 3、Phase 4 的结果描述中统一口径。

建议新增文案：

```markdown
- **集合职责**：训练集用于分类器训练；验证集仅用于阈值调优、模型选择和早停；测试集仅用于最终一次性能汇报与主结果图表生成，避免将调参结果误写为最终结果。
```

同时建议将类似表述统一改成：

```markdown
PPL 方法在验证集上调优阈值，并在测试集上汇报最终结果
SAPLMA 在验证集上完成模型选择，并在测试集上汇报最终结果
```

### P1. Windows 下 bitsandbytes 的推荐安装路径仍然前后不一致

当前计划已经把 bitsandbytes 降为可选依赖，这是对的；但文档前部的安装步骤仍写成：

```powershell
uv add bitsandbytes
```

而后面的 Windows 风险部分又建议使用 Windows 兼容 wheel。这会让执行者在 PowerShell 环境下不知道哪条才是推荐路径。

影响：

- 同一份文档里出现两套不同的 Windows 安装方案；
- 新成员可能先安装标准包，再在兼容性问题出现后回头修补；
- 与“Qwen2-1.5B 先保底跑通”的主线不完全一致。

建议：前面的依赖安装部分直接改成“仅在尝试 Llama-2-7B 4-bit 且确认采用 Windows 兼容 wheel 时安装”；或者把标准命令删掉，只保留风险章节中的 Windows 推荐方案。

建议替换文案：

```powershell
# 仅在尝试 Llama-2-7B 4-bit 且需要 Windows 兼容安装时使用
uv add https://github.com/jllllll/bitsandbytes-windows-webui/releases/download/wheels/bitsandbytes-0.43.3-py3-none-win_amd64.whl
```

如果你们希望保留跨平台写法，也建议补一句：

```markdown
注：本计划默认执行环境为 Windows PowerShell，因此 bitsandbytes 在本文中优先采用 Windows 兼容安装方案；标准 `uv add bitsandbytes` 仅作为非 Windows 环境参考。
```

### P2. 注意力方向中的“关系词 token”仍缺少可复现的抽取规则

Phase 4 已经把注意力特征改写成更适合陈述句的实体-关系注意力，这是正确的；但当前文档只明确了 Subject token 的抽取方式，并没有定义“关系词 token”如何定位。

如果这一点继续保持模糊，P4.A2 仍会面临两个问题：

- 不同成员可能用不同的人工理解方式标注关系词；
- 同一句陈述在不同抽取规则下可能得到不同注意力特征，难以复现。

建议：至少给出一个保底规则，并说明失败时的退化方案。比如：

1. 优先使用依存句法中与主语实体关联最紧密的谓词或系词短语作为关系词 span；
2. 若自动解析失败，则退化为主语后第一个核心动词或系表结构；
3. 若句法结构过于复杂，可将该样本排除在关系词特征分析之外，并在报告中说明比例。

建议新增文案：

```markdown
- **关系词 token 提取**：在注意力实验中，优先使用依存句法解析得到与主语实体对应的核心谓词或系词短语作为关系词 span；若自动解析失败，则退化为主语后第一个核心动词（或系表结构）；若规则仍不稳定，则仅保留主语实体与句尾 token 的注意力特征，并在报告中说明。
```

### P2. 复现要求还需要和锁定文件显式绑定

当前文档已经强调“可比性与可复现性”，也列出了 [docs/Project_Plan.md](docs/Project_Plan.md) 中的 `environment.yml`、`pyproject.toml` 和 `uv.lock`。但计划正文还没有明确写出：最终提交和复现实验时，应以哪些文件作为环境的准确定义。

如果这一点不写明，软件环境表里的宽版本范围仍可能让不同成员理解成“这些版本附近都可以”。这在讨论原则上没问题，但和“最终可复现提交”不是同一层意思。

建议：在环境或 Phase 5 中补一句，明确最终提交以锁定文件为准，环境表只表达原则范围。

建议新增文案：

```markdown
**复现约定**：软件环境表中的版本范围用于说明依赖原则；实际提交与复现实验时，以 `environment.yml`、`pyproject.toml` 和 `uv.lock` 中锁定的环境配置为准。
```

## 4. 修改优先级建议

如果只再做一轮最小修订，建议按以下顺序处理：

1. 先补清验证集与测试集的职责边界；
2. 再统一 Windows 下 bitsandbytes 的唯一推荐安装路径；
3. 接着补上关系词 token 的抽取规则；
4. 最后把复现要求和锁定文件绑定起来。

## 5. 一句话总结

这份计划已经接近“成熟执行版”了；v3 剩下的更多是实验闭环和落地细节。如果把测试集职责、Windows 依赖路径、关系词抽取规则和锁定文件约定再补齐，后续基本就可以把审查重心从“修文档”转向“盯实现”。
