## 4. 实践要点

### 4.1 明确两类基线方法的接口边界

Phase 2 的关键不是只“把代码写出来”，而是要把两个基线方法的职责拆清楚：

- PPL 方法负责生成连续分数，并通过阈值转为真 / 假判别
- SAPLMA 方法负责将隐藏状态映射为监督分类问题

因此需要分别设计：

- 分数函数
- 特征提取函数
- 分类训练函数
- 预测与概率输出函数
- 统一评估入口

### 4.2 层索引语义必须与文档保持一致

文档明确要求：

- 层号按 Transformer block 输出统计
- 不把 embedding output 计入层号

这意味着在隐藏状态实现中，必须显式处理：

- `hidden_states[0]` 为 embedding output 的情况
- 某些测试桩或兼容场景下不含 embedding output 的情况
- 负索引与正索引都要有清晰语义

这是后续 Phase 3 层分析能否稳定开展的前提。

### 4.3 PPL 阈值方向不能搞反

PPL 不是“越高越真”，而是：

- **PPL 越低，模型越认可该陈述**
- 因此搜索最优阈值时，要么在评估时使用 `higher_is_positive=False`
- 要么在阈值搜索阶段先对分数取负再优化

这一点属于 Phase 2 中非常关键的实现细节。

### 4.4 测试不仅验接口，还要验边界

在 Phase 2 中，测试不能只停留在“函数存在”。

还需要验证：

- dummy model 与真实模型都能工作
- 批量与单样本逻辑一致
- 返回结果维度稳定
- 分数顺序与输入顺序一致
- 非法参数能抛出明确异常
- 模型缺失某些能力时存在合理退化路径

这种做法有助于为后续 Phase 3 / Phase 4 的扩展提供更稳固的基础。

### 4.5 延迟导入降低环境噪声影响

在真实运行中，Windows 环境下某些原生依赖（如 `pandas` / `pyarrow`）存在初始化噪声风险。

因此在数据模块中采用按需导入策略，可以减少：

- 不必要的导入副作用
- 仅做接口测试时触发原生依赖初始化的问题

这是一种工程化稳健性优化，而不仅仅是功能实现。
